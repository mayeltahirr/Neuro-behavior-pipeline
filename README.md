# Neural Activity + Behavior Decoding Pipeline

A small pipeline that takes simulated neural recordings and animal behavior
data, extracts firing-rate features, trains a model to decode behavior from
neural activity, and stores everything in a queryable research database.

## Why synthetic data

I don't have access to real neurophysiological recordings, so everything
in `data/` and generated by `src/simulate.py` is synthetic -- built to have
the same *shape* as data a lab would actually collect: spike trains from a
population of neurons, and a behavior ethogram (a timeline of what the
animal was doing: resting, grooming, or moving). Firing rates are
deliberately tied to behavior state -- some neurons care about "move," some
about "groom," some don't care much at all -- the way a real population
usually has a mix of tuned and untuned cells. That's what lets the model
actually find a signal instead of learning off pure noise.

I built this to demonstrate the kind of pipeline work a computational
neuroscience role involves: turning large, messy recordings into curated,
analyzable structure, and applying a statistical model on top -- not to
claim I've already worked with real large-scale neural or behavioral
datasets.

## What's in here

```
raw spikes + ethogram --> features.py --> model.py --> repository.py
        (simulate.py)      (firing rate      (decode        (store in
                             per time bin)     behavior)      SQLite)
```

- **`src/simulate.py`** -- generates a synthetic recording session: spike
  trains for N neurons, and a behavior ethogram splitting the session into
  bouts of rest, groom, or move.
- **`src/features.py`** -- bins the session into fixed time windows (1s by
  default) and computes each neuron's firing rate per bin, labeled with
  whichever behavior was happening.
- **`src/model.py`** -- trains a logistic regression classifier to predict
  the behavior label from population firing rates, and reports held-out
  accuracy.
- **`src/repository.py`** -- writes the session, the ethogram, and the
  model's results into a small SQLite database (`sessions`,
  `behavior_bouts`, `model_results` tables) -- the "data curation and
  storage in a research repository" part.
- **`src/pipeline.py`** -- runs all of the above in order from the CLI.
- **`data/`** -- a small example of what the synthetic data actually looks
  like (`example_behavior_bouts.csv`, `example_spike_times.csv`), so you
  can see the shape of it without running anything first. See
  `data/README.md` for details -- everything in there is synthetic, not
  from a real animal.

## Running it

```bash
pip install -r requirements.txt
python src/pipeline.py --neurons 20 --duration 600
```

Example output:
```
Simulating a 600s session with 20 neurons...
Extracting firing-rate features per time bin...
Training a classifier to decode behavior from firing rates...
Held-out accuracy: 91.33%  (chance level for 3 behaviors is ~33%)
Storing session, ethogram, and results in research_repository.sqlite...
Done. session_id=1
```

That accuracy is well above chance because the synthetic data is built with
a real behavior-firing rate relationship baked in -- the point is to show
the pipeline can actually recover that signal, not to claim state-of-the-art
decoding performance.

You can then query the repository directly:
```bash
sqlite3 research_repository.sqlite "SELECT * FROM sessions;"
sqlite3 research_repository.sqlite "SELECT behavior, COUNT(*) FROM behavior_bouts GROUP BY behavior;"
```

## Tests

```bash
pytest tests/ -v
```

These check the spike-binning and behavior-labeling logic against small,
hand-built spike arrays -- fast, and easy to verify by eye, no simulated
session or database required.

## Troubleshooting

A few things I'd expect to trip someone up running this, based on what
the code actually does (rather than a debugging story -- it ran clean for
me the first time, so take this as "things to know," not "things that
happened"):

- **`pip: command not found`** -- same as any Python project on a fresh
  Mac, use `pip3` or `python3 -m pip install -r requirements.txt` instead.
- **`ValueError: This solver needs samples of at least 2 classes`** --
  happens if `--duration` is too short to contain more than one behavior
  bout, so the training set ends up all one label. I actually ran
  `--neurons 5 --duration 20` to check this and hit it immediately.
  Keep `--duration` at 300+ seconds and it won't come up.
- **`sqlite3.OperationalError: database is locked`** -- only really
  shows up if you have `research_repository.sqlite` open in another
  program (like DB Browser for SQLite) while the pipeline runs. Close it
  first, or point `--db` at a different file.
- **Accuracy looks suspiciously high or low** -- the accuracy is directly
  tied to how strongly `_neuron_rate_profile()` ties firing rate to
  behavior. If you change those numbers in `simulate.py`, expect the
  model's accuracy to move accordingly -- that's the synthetic data
  working as intended, not a bug.

## Where I'd take this next

- Right now the "behavior" is a single label per time bin. A real ethogram
  often has overlapping or hierarchical behaviors, which would need a
  different labeling approach.
- The model is a simple logistic regression on raw firing rates. A next
  step would be trying a model that uses spike timing structure directly
  (e.g. a GLM per neuron, or a sliding-window CNN) rather than binned
  rates.
- No optogenetic or transcriptomic data here yet -- those would need their
  own simulation logic and a different set of repository tables.
